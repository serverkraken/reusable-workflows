# Phase 9.5 — Failure-Paths Cron (Design Spec)

**Datum:** 2026-05-27
**Quelle:** Soenne-Feedback auf PR #144 ("Das Ziel wurde nicht erreicht"). Phase 9 (PR #143, v4.3.0) implementiert Summary-Job-Aggregation, erreicht damit aber nicht den **eigentlichen** Designaspekt der Spec: einen visuell-grünen Top-Level-PR-Status-Badge.
**Scope:** Verschiebt alle 11 `test-*-fail` / `assert-*-fail` Job-Pairs aus `self-ci.yml` + `integration.yml` in einen neuen nightly-cron `failure-paths-nightly.yml`. Bündelt das ursprünglich für Phase 9b geplante Cleanup (22 `caller-*.yml` löschen + Goreleaser-Fixture-Fix).
**Konsumiert von:** Implementation Plan (writing-plans als Nachfolger)
**Vorgänger:** Phase 9 (PR #143, v4.3.0)

---

## 1. Goal

Auf einer beliebigen PR-Page des Catalog-Repos zeigt der Top-Level-Status-Badge **grün**, sobald alle Atom-Happy-Paths sowie die zwei Summary-Jobs durchlaufen. Designed-failure-Path-Tests laufen weiterhin gegen die echten Atome (echte `uses:` mit broken Fixtures), aber **außerhalb des PR-Check-Rollups** — nightly um 04:00 UTC.

Bei einer Regression in einem Failure-Path (Atom-Logik geändert, Fixture stale, Action-Version-Drift) öffnet der nightly-Workflow automatisch ein Issue mit Label `failure-path-regression` und kommentiert auf bereits offene Issues statt zu duplizieren.

## 2. Background: Warum Phase 9 das Ziel verfehlt hat

Phase 9 spec Section 1 behauptete: *"Der Top-Level-PR-Status-Badge wird grün auch wenn intern designed-red-Jobs laufen, weil Summary nur die `assert-*` + happy-Jobs in `needs:` aufnimmt."*

Diese Behauptung ist konzeptuell falsch:

| Mechanismus | Was es kontrolliert |
|-------------|---------------------|
| Branch-Protection-Required-Checks | Mergebarkeit (Block-on-Red) |
| PR-Status-Rollup (visuell) | Aggregation **aller** Checks für den SHA — ohne Filter auf required-Liste |

PR #144 demonstrierte das in der Praxis: 11 `test-*-fail`-Sub-Job-Checks blieben rot im Rollup obwohl beide Summary-Checks grün waren und keine required-Checks gesetzt waren. GitHub's UI zeigt "Some checks have failed" sobald irgendein Check rot ist.

**Die strukturelle Konsequenz:** Solange Failure-Path-Atome via `uses:` aufgerufen werden, exposen sie ihre internen Jobs als eigene PR-Checks. Es gibt keinen Workflow-Level-Mechanismus, einen completed Job aus dem Rollup auszublenden. Die einzige Lösung ist, die Atom-Aufrufe **nicht zur PR-Time** stattfinden zu lassen.

### Out of Scope

- **Slack-Integration** für Regression-Notifications. Standardmäßig nur GitHub-Issue. Slack-Webhook kann später als zusätzlicher Step nachgerüstet werden, wenn relevant. YAGNI für jetzt.
- **Multi-Stage-Issue-Lifecycle** (auto-close bei nächstem grünem run). Issues bleiben offen bis ein Mensch sie schließt — das stellt sicher dass ein gefixter Regression nicht still gewipt wird ohne Review.
- **Per-Atom-Granularity in Issue-Title.** Issue-Title trägt das Datum + Liste der gefailten Asserts. Mehr ist unnötig wenn die Issue-Body Run-URL hat.
- **Cron-Frequenz erhöhen.** Daily ist hinreichend für Catalog-Repo-Velocity. Wenn ein Atom-Change auf main mergt und das Failure-Verhalten regressiert, sieht Soenne das spätestens am nächsten Morgen.
- **Smart-Notification (nur bei Status-Change).** Aktuell: jedes nightly mit irgendeiner Failure → kommentiert auf bestehende Issue. Idempotent, aber lautstark wenn 5 Tage hintereinander dasselbe regressiert. Acceptable Trade-off.
- **runs_on-Override für nightly auf ubuntu-latest.** Atome behalten ihren default `runs_on` (typischerweise `[self-hosted, Linux]`). Cosmetic-consistency mit PR-time Verhalten.

## 4. Architecture

### 4.1 `failure-paths-nightly.yml` Aufbau

```yaml
name: failure-paths-nightly

on:
  schedule:
    - cron: '0 4 * * *'   # Daily 04:00 UTC
  workflow_dispatch:

permissions:
  contents: read
  issues: write
  actions: read
  # Atom-spezifische Permissions werden via secrets: inherit + nested permissions
  # vom Atom selbst gefordert. Hier nur was der nightly's eigene Logik braucht.

concurrency:
  group: failure-paths-nightly
  cancel-in-progress: false   # never cancel a running nightly

jobs:
  # ─────────────────────────────────────────────────────────────
  # LINT FAIL ATOMS (4)
  # ─────────────────────────────────────────────────────────────
  test-lint-go-fail:
    uses: ./.github/workflows/lint-go.yml
    secrets: inherit
    with:
      working_directory: tests/fixtures/lint-test/go-lint-fail

  assert-lint-go-fail:
    needs: test-lint-go-fail
    if: always()
    runs-on: ubuntu-latest
    steps:
      - name: Assert lint job failed
        env:
          R: ${{ needs.test-lint-go-fail.result }}
        run: |
          if [[ "$R" != "failure" ]]; then
            echo "::error::expected failure, got: $R"
            exit 1
          fi
          echo "lint-go-fail: correctly observed failure"

  # ... gleiche Struktur für lint-helm-fail, lint-python-fail, lint-rust-fail

  # ─────────────────────────────────────────────────────────────
  # COV-FAIL ATOMS (3)
  # ─────────────────────────────────────────────────────────────
  # test-test-go-cov-fail, test-test-python-cov-fail, test-test-rust-cov-fail
  # plus assert-* siblings

  # ─────────────────────────────────────────────────────────────
  # SIDE-EFFECTS FAIL ATOMS (4)
  # ─────────────────────────────────────────────────────────────
  # test-cleanup-images-fail, test-docker-build-multi-fail,
  # test-goreleaser-fail, test-helm-publish-fail
  # plus assert-* siblings

  # ─────────────────────────────────────────────────────────────
  # REGRESSION REPORTER
  # ─────────────────────────────────────────────────────────────
  report-regressions:
    needs:
      - assert-lint-go-fail
      - assert-lint-helm-fail
      - assert-lint-python-fail
      - assert-lint-rust-fail
      - assert-test-go-cov-fail
      - assert-test-python-cov-fail
      - assert-test-rust-cov-fail
      - assert-cleanup-images-fail
      - assert-docker-build-multi-fail
      - assert-goreleaser-fail
      - assert-helm-publish-fail
    if: always() && contains(needs.*.result, 'failure')
    runs-on: ubuntu-latest
    permissions:
      issues: write
    steps:
      - name: Open or comment on regression issue
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NEEDS_JSON: ${{ toJson(needs) }}
          RUN_URL: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}
        run: |
          set -euo pipefail
          failed=$(echo "$NEEDS_JSON" | jq -r '
            to_entries
            | map(select(.value.result != "success"))
            | .[].key
          ' | paste -sd ',' -)
          today=$(date -u +%Y-%m-%d)

          existing=$(gh issue list \
            --label failure-path-regression \
            --state open \
            --limit 1 \
            --json number \
            --jq '.[0].number // empty')

          if [[ -n "$existing" ]]; then
            gh issue comment "$existing" --body "Nightly $today — failed asserts: \`$failed\`. Run: $RUN_URL"
          else
            gh issue create \
              --label failure-path-regression \
              --title "[failure-path-regression] nightly $today: $failed" \
              --body "$(cat <<EOF
          Nightly \`failure-paths-nightly.yml\` ran on $today and one or more failure-path asserts did not observe the expected failure.

          **Failed asserts:** \`$failed\`
          **Run:** $RUN_URL

          **Likely causes:**
          - Atom logic was refactored and no longer fails on the broken fixture
          - Fixture is stale (action version updated, behavior changed)
          - Action version drift (e.g., goreleaser-action auto-generates default config)

          **Triage checklist:**
          - [ ] Identify which atom/fixture pair regressed (see failed-asserts list above)
          - [ ] Inspect recent commits to the named atom workflow
          - [ ] Verify the fixture still triggers the failure mode under the current atom logic
          - [ ] Repair atom OR fixture OR remove the failure-path test if obsolete
          - [ ] Close this issue with a link to the fixing PR
          EOF
          )"
          fi
```

### 4.2 Why nightly-Runs nicht im PR-Rollup auftauchen

GitHub PR check rollups zeigen Workflow-Runs für den PR-SHA mit Trigger-Events aus `{pull_request, push}` und ähnlichen PR-relevanten Quellen. Cron-getriggerte Runs (`schedule` Event) laufen gegen den default-branch (typischerweise main), nicht gegen einen PR-SHA, und erscheinen ausschließlich im Repo-Level Actions-Tab.

`workflow_dispatch` ebenfalls nicht im PR-Rollup, da der Run nicht an einen PR-SHA gebunden ist. Manuelle Trigger des nightly bleiben so im Actions-Tab sichtbar ohne PRs zu kontaminieren.

### 4.3 Issue-Lifecycle

```
[nightly run]
  → all asserts succeed → no issue action
  → ≥1 assert fails:
     → existing open issue with label `failure-path-regression`? → comment auf existing
     → kein existing → create new with full triage body
[human fixes the regression]
  → next nightly run all green → no automatic close
  → human closes issue manually with link to fixing PR
```

Rationale: Auto-close ist riskant, weil ein Single-success-Run nicht beweist dass die Regression wirklich behoben ist (könnte intermittent sein). Manuelle Close + Fix-PR-Link in der Issue-Schließungs-Notiz schafft saubere Audit-Trail.

### 4.4 Concurrency

`concurrency.group: failure-paths-nightly` (kein `${{ github.ref }}` Suffix) — verhindert dass zwei nightlies sich überlappen. `cancel-in-progress: false` — nightly mid-flight wird nie abgebrochen, auch wenn ein `workflow_dispatch` mitten in einer schedule-run ausgelöst wird.

### 4.5 Permissions

`failure-paths-nightly.yml` top-level Block:

```yaml
permissions:
  contents: read     # for checkout in the regression-reporter
  issues: write      # for issue creation/comment
  actions: read      # for the run URL in issue body (cosmetic, not strictly needed)
```

Atome erben ihre permissions via `secrets: inherit`. Es ist möglich dass einzelne Atome zusätzliche permissions am Wrapper-Level brauchen (z. B. `packages: write` für docker-build-multi mit `push: true` — aber nightly nutzt explizit `push: false`/`prerelease: true` Patterns aus den heutigen caller-* Definitionen). Audit beim Atom-Import in Task X des Plans.

### 4.6 Triggers

- `schedule: 0 4 * * *` — täglich 04:00 UTC (06:00 CEST, 05:00 CET)
- `workflow_dispatch:` — manual override, z. B. nach Atom-Refactor um Failure-Path-Coverage sofort zu verifizieren ohne 24h zu warten

## 6. Rollback

PR-Revert via `git revert <merge-commit>` reverts:
- 11 Job-Pairs zurück in Wrapper
- 22 caller-*.yml zurück im Repo
- Goreleaser-Fixture zurück auf leer (kein .goreleaser.yaml)
- Branch-Protection: muss manuell zurück (Required-Checks-Liste vor 9.5 sichern via `gh api ...protection --jq '.required_status_checks'` bevor Update läuft)

Issue-Label `failure-path-regression` bleibt erhalten — harmless.

`failure-paths-nightly.yml` wird beim revert gelöscht; bestehende cron-runs sind dann obsolete und brechen ab.

## 7. Testing

### PR-Time Validation
- `gh pr checks --watch` — alle Checks müssen grün sein, KEIN test-*-fail oder assert-*-fail im Rollup
- actionlint passes
- `goreleaser check` lokal in `tests/fixtures/cli-go-no-config/` exitet non-zero

### Nightly Workflow Validation
- Vor PR-Merge: PR-Branch via `gh workflow run failure-paths-nightly.yml --ref feat/phase-9-5-failure-paths-cron` manuell triggern (workflow_dispatch funktioniert auf branches die den workflow tragen)
- Beobachten dass alle 11 assert-*-fail grün sind
- Beobachten dass `report-regressions` skipped wird (kein contains-failure)

### Post-Merge Validation
- 24h nach merge: ersten cron-getriggerten run im Actions-Tab prüfen
- Falls alle grün: keine Issue erstellt — Phase 9.5 funktioniert
- Falls eine assert rot: ein Issue mit Label `failure-path-regression` erstellt — System funktioniert wie spezifiziert

### Regression-Reporter Probe (optional in PR)
- Test mit deliberately-success-fixture in einem der test-*-fail Jobs (z. B. `go-happy` statt `go-lint-fail` im lint-go-fail Job)
- Beobachten dass `report-regressions` aktiviert wird und ein Issue erstellt
- Vor PR-Push: zurückrollen

## 8. Risiken

| Risiko | Wahrscheinlichkeit | Impact | Mitigation |
|--------|---------------------|--------|------------|
| GitHub-Cron-Drift (cron-runs werden nicht zuverlässig getriggert bei niedriger repo-activity) | niedrig | mittel | `workflow_dispatch` als manueller Backup; Onboard-Sweep läuft ohnehin täglich und hält das Repo aktiv |
| Issue-Spam wenn Regression über Tage nicht gefixt | mittel | niedrig | Single-issue-pattern via `gh issue list --label failure-path-regression --state open --limit 1` — Kommentiert auf existing statt zu duplizieren |
| Goreleaser-Fixture trotz broken-YAML noch success | niedrig | niedrig | Lokal mit `goreleaser check` validieren vor Push; Plan dokumentiert Fallback-Inhalte (z. B. `version: -1`) |
| Branch-Protection-Update post-merge vergessen | mittel | niedrig | gh-api Snippet in PR-Beschreibung als Checklist-Item; Memory `phase_workflow_pattern` enforced 2-stage review |
| `report-regressions` failt selbst (jq-syntax, gh-CLI-permissions) und das Failure ist still | niedrig | mittel | `set -euo pipefail` im Step-Script; Step-Failure ist im Actions-Tab als rote Run sichtbar; PR-time Validierung durch workflow_dispatch run |
| Atom-Permissions in nightly nicht erfüllt (z. B. integration.yml hatte top-level permissions die nightly nicht hat) | mittel | mittel | Audit der erforderlichen permissions pro Atom in Task 5 des Plans; Union an top-level permissions ergänzen |
| Label `failure-path-regression` existiert nicht im Repo | mittel | niedrig | `gh issue create --label` failed mit klarem error; Plan Task 17 erstellt das Label einmalig vor erstem nightly |
